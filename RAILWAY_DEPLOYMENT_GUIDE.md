# PSM Enterprise — Railway Deployment Guide

## ⚠️ Understanding the Issue

Your project deployment failed because:
- **railway.json** was configured to use a "dockerfile" builder
- Railway doesn't support `docker compose` as build/start commands
- Your project uses multiple services (backend, frontend, DB, Redis) that need coordination

---

## ✅ Recommended Solution: Multi-Service Railway Deployment

Railway supports deploying multiple services using their dashboard. This is the **best approach** for your architecture.

### Option 1: Deploy Each Service Separately (RECOMMENDED)

#### Step 1: Set Up PostgreSQL Database
1. In Railway dashboard → **New Project** → **Add Service** → **Database** → **PostgreSQL**
2. Railway automatically creates the database
3. Copy the connection string from the service details

#### Step 2: Set Up Redis Cache
1. Add another service → **Database** → **Redis**
2. Copy the Redis connection URL

#### Step 3: Deploy Backend (C# ASP.NET Core)
1. Add service → **GitHub Repo** (connect your repo)
2. Set the **Dockerfile**: `Dockerfile.backend-csharp`
3. Add environment variables:
   ```
   ASPNETCORE_ENVIRONMENT=Production
   ASPNETCORE_URLS=http://+:3000
   PORT=3000
   PG_HOST=<postgres-hostname>
   PG_PORT=5432
   PG_DATABASE=railway
   PG_USER=postgres
   PG_PASSWORD=<your-postgres-password>
   REDIS_CONNECTION=redis://default:<redis-password>@<redis-hostname>:6379
   JWT_SECRET=<generate-64-char-random-string>
   CORS_ORIGINS=https://<your-frontend-domain>.railway.app
   ADMIN_SEED_USERNAME=admin
   ADMIN_SEED_PASSWORD=<secure-password>
   PSM_LICENSE_SECRET=<generate-random-string>
   ANTHROPIC_API_KEY=<optional-if-using-AI>
   ```
4. Set **Start Command**: Leave empty (uses Dockerfile ENTRYPOINT)
5. Set **Build Command**: Leave empty (uses Dockerfile)

#### Step 4: Deploy Frontend (React + Nginx)
1. Add service → **GitHub Repo**
2. Set the **Dockerfile**: `Dockerfile.frontend`
3. Build args:
   ```
   VITE_API_URL=https://<your-backend-domain>.railway.app/api
   ```
4. Set port mapping: `80:80`
5. Add environment variable:
   ```
   VITE_API_URL=/api
   ```

#### Step 5: Connect Services
In Railway dashboard → **Networking**:
- Link Frontend → Backend (so frontend can reach backend API)
- Link Backend → PostgreSQL (database connection)
- Link Backend → Redis (cache connection)

---

### Option 2: Deploy as Single Container (Alternative)

If you prefer a single container approach, you'll need a wrapper script.

#### Create a Docker Entrypoint Script

Create `docker-entrypoint.sh`:
```bash
#!/bin/bash

# Start PostgreSQL (requires postgres in base image)
# Note: This approach requires a custom base image - not recommended

# Start Redis
redis-server --daemonize yes

# Start Backend
dotnet PSM.Api.dll &

# Wait for backend to be ready
sleep 10

# Start Frontend (Nginx)
nginx -g "daemon off;"
```

**This approach is complex** and not recommended because:
- Requires managing multiple processes in one container
- Difficult to restart individual services
- Hard to scale each component independently
- Railway's built-in service management is better

## 🔧 Environment Variable Setup

### Critical Variables (REQUIRED)
```bash
# Database (from Railway PostgreSQL service)
PG_HOST=<hostname>
PG_PORT=5432
PG_DATABASE=railway
PG_USER=postgres
PG_PASSWORD=<auto-generated>

# Redis (from Railway Redis service)
REDIS_CONNECTION=redis://default:<password>@<hostname>:6379

# Application
JWT_SECRET=<must-be-64-chars-minimum>
JWT_EXPIRES_IN=24h
ASPNETCORE_ENVIRONMENT=Production

# Admin account (for first login)
ADMIN_SEED_USERNAME=admin
ADMIN_SEED_PASSWORD=<secure-password>

# CORS (adjust to your domains)
CORS_ORIGINS=https://your-frontend.railway.app,https://your-domain.com

# License
PSM_LICENSE_SECRET=<generate-random-string>
```

### Optional Variables
```bash
# AI Integration (optional)
ANTHROPIC_API_KEY=<your-anthropic-key>
AI_MODEL=claude-sonnet-4-6
AI_ENABLED=true

# Application Settings
PORT=3000
VITE_API_URL=/api
```

## 🚀 Deployment Steps

### Step-by-Step Guide

1. **Create Railway Account**
   - Go to [railway.app](https://railway.app)
   - Sign up with GitHub

2. **Create New Project**
   - Click "New Project"
   - Select "Deploy from GitHub repo"

3. **Add PostgreSQL Service**
   - Click "Add Service"
   - Select "Database" → "PostgreSQL"
   - Railway creates it automatically
   - Note the connection details

4. **Add Redis Service**
   - Click "Add Service" again
   - Select "Database" → "Redis"
   - Note the connection URL

5. **Deploy Backend Service**
   - Add Service → GitHub repo (select your fork)
   - Configure:
     - **Dockerfile**: `Dockerfile.backend-csharp`
     - **Environment**: Add all PG_ and REDIS_ variables
     - **Port**: 3000
   - Railway auto-detects and builds

6. **Deploy Frontend Service**
   - Add Service → GitHub repo (same repo)
   - Configure:
     - **Dockerfile**: `Dockerfile.frontend`
     - **Environment**: Add VITE_API_URL pointing to backend domain
     - **Build Args**: `VITE_API_URL=/api`
     - **Port**: 80
   - Railway auto-detects and builds

7. **Link Services**
   - Go to project settings → **Networking**
   - Create connections between services
   - Frontend needs access to Backend API
   - Backend needs access to PostgreSQL and Redis

## 🔐 Security Best Practices

1. **JWT_SECRET**: Generate a strong 64+ character string
   ```bash
   # Generate strong secret
   node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
   ```

2. **Database Password**: Railway auto-generates, but ensure it's strong

3. **CORS_ORIGINS**: Only list your actual domains
   ```
   https://your-app.railway.app
   https://www.yourdomain.com
   ```

4. **Admin Password**: Use a strong, unique password
   - Change it immediately after first login
   - Don't commit to version control

5. **API Keys**: Use Railway's secret management
   - Add sensitive values in **Variables** (marked as secret)
   - They won't appear in logs or build output

## ❌ Common Deployment Errors & Fixes

### Error: "Deployment failed during build process"
**Cause**: Dockerfile or build step failing

**Solution**:
- Check Railway logs (Deployments tab)
- Verify Dockerfile path is correct
- Ensure all dependencies are available
- Check NuGet timeout (set to 300s)

### Error: "No database found when starting backend"
**Cause**: Backend service starting before PostgreSQL is ready

**Solution**:
- Add healthcheck to PostgreSQL service in Railway
- Set proper startup dependencies
- Use Railway's built-in service linking

### Error: "Frontend can't reach backend API"
**Cause**: CORS or networking issue

**Solution**:
- Set CORS_ORIGINS to include frontend domain
- Verify VITE_API_URL points to correct backend URL
- Check Firebase -> Networking in Railway dashboard

### Error: "502 Bad Gateway"
**Cause**: Backend not responding or health check failing

**Solution**:
- Check backend logs in Railway
- Verify database connection string
- Confirm all environment variables are set
- Check port configuration (should be 3000)

## 🔄 Database Migrations

Railway PostgreSQL service starts fresh. Migrations run automatically when backend starts:

1. Backend service starts
2. Connects to PostgreSQL
3. Runs all migrations from `/backend-csharp/Migrations/` directory
4. Creates schema and seed data
5. Admin account created with credentials you provided

**No manual setup needed** - migrations run automatically!

## 🌐 Custom Domains

To add your own domain:

1. In Railway project settings → **Domains**
2. Add new domain (e.g., `app.yourdomain.com`)
3. Update DNS records (CNAME to Railway's provided value)
4. Update CORS_ORIGINS environment variable
5. Redeploy backend service

## 📈 Scaling & Performance

### Default Configuration (Free Tier)
- 0.5 CPU, 256MB RAM per service
- May be slow on startup
- Good for development/testing

### Upgrade Configuration (Recommended)
- Increase Backend: 1 CPU, 512MB RAM
- Increase Frontend: 0.5 CPU, 256MB RAM
- Increase Database: 1 CPU, 512MB RAM
- Increase Redis: 0.5 CPU, 256MB RAM

Click service → **Resources** to adjust

## 🆘 Support & Troubleshooting

**Check Logs First**: Most issues are visible in Railway logs
- Look for error messages with timestamps
- Check build step errors
- Verify environment variable substitution

**Common Issues**:
1. **Build timeout**: Increase NuGet timeout in Dockerfile
2. **Memory issues**: Increase service resource allocation
3. **Connection refused**: Check service linking and ports
4. **Migration failures**: Check database logs for schema conflicts

**Contact**: Create an issue in your repository with:
- Deployment error screenshot
- Railway logs export
- Your railway.json configuration
- Environment variables list (without secrets)
